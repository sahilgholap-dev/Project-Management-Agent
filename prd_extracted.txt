NEXUS PM Agent
Product Requirements Document — v5
Scope: one client running multiple concurrent projects. Timeline tracked separately.
1. Overview & Scope
NEXUS PM Agent automates the operational work of project management — planning, tracking, risk management, governance, and stakeholder communication — for a single client running one or more concurrent projects.
Correction from v2:  v2 implied a multi-tenant SaaS product serving many separate client organizations, with a full control-plane and schema-per-tenant isolation layer built up front. Scope was corrected to what is actually being built first: one client, multiple projects, sharing one schema. Multi-tenant support is documented as a deferred future option (Section 14), not built now.

Correction from v4:  Fable 5 and Sonnet 5 serve different roles and should not be conflated. Fable 5 is the build-time tool — used through Claude Code for the long-horizon, autonomous work of writing this system. At real usage volume, running every routine skill call on Fable 5 pricing would be paying for autonomy the runtime doesn't need.
3. Doctrine: Config, Not Code
One codebase serves every project. All project-specific behavior lives in configuration, not in project-specific code branches. If serving a new project ever requires a code change rather than a config change, that is a defect in the doctrine, not a valid one-off exception.

Admin flow
Admin login → admin dashboard → create company → create user for that company → send invite → setup complete.
User flow
User receives invite → user login → configure company (writes client_config) → create teams → set reviewer & escalation rules → save configuration → agent goes live.
Project intake
Submitted per project, not once per client. Each new project triggers ingestion independently and lands as a new row in projects, sharing the client's config defaults unless a project-specific override is set.
5. Client & Project Configuration
client_config — client-level defaults
About the client, project definition, cadence, skill depth per skill, tools/channels, primary reviewer, backup reviewer, escalation delay, change approver, sign-off approver, voice/style, working calendar, assignment strategy, and slip-flag threshold.
projects.config_overrides — per-project overrides
Any project may override any client-level default — for example, a different reviewer or a tighter escalation delay on one high-visibility project. Resolution order at read time: project override first, client default otherwise.
Correction from v2:  Three fields were added following the v2 review that were previously missing from client_config: working_calendar, assignment_strategy, and slip_threshold_days.
Correction from v2:  Escalation delay was a single value applied identically to all tiers in v2. An optional escalation_delay_by_tier override was added, since a minor internal flag and a client-facing artifact do not warrant the same wait before escalating.
6. Project Structure: Phases & Tasks
Task breakdown runs in two passes, not one. First, the agent breaks the project's scope into phases — each with its own description, timeline, and status. Only then does it generate the detailed tasks within each phase, along with that phase's documentation.

8. Agent Skill Flows (Detailed)
Full trigger-to-output sequence for each of the eight skills in Section 7. Previously a separate companion document; merged in here as of v5 so the skill library and its exact behavior stay in one place.
8.1  Task Breakdown
  LLM reasoning — Claude Sonnet 5  
Trigger:  New project intake, or an admin-requested re-breakdown
Reads:  projects.scope_summary, budget_total, timeline_start/end
Writes:  phases, tasks, task_dependencies
Runs in two passes — project → phases first, then each phase → its own detailed tasks. Never generates tasks directly from the raw scope document.
1.  Load the project's scope summary, budget, and timeline from projects.
2.  Send the scope document to Sonnet 5 with an instruction to decompose it into phases only — name, description, and a rough date range per phase.
3.  Sonnet 5 returns a structured phase list. Anything ambiguous in the scope is flagged as needs_clarification rather than guessed.
4.  Validate the phase output against the schema. A missing required field halts here and surfaces to the reviewer — it does not get silently patched.
5.  Write validated phases into the phases table, in sequence order.
6.  For each phase in turn, send that phase's scope back to Sonnet 5 with an instruction to decompose it into detailed tasks: title, effort estimate, required skill tags, and any dependency mentioned in prose.
7.  Validate the task output. Sum effort estimates for the phase and check that total against the phase's date range and the client's working_calendar — flag to the reviewer if the plan is already over capacity before a single task starts.
8.  Write validated tasks into tasks, each linked to its phase_id.
9.  Record any dependency mentioned between tasks as a row in task_dependencies.
10.  Hand off to the Scheduler (8.2) to compute actual dates from this task list.
11.  Log the full ingestion-and-breakdown action to audit_log, with the scope document as input and the phase/task list as output.
8.2  Scheduler
  Deterministic — plain code, no LLM  
Trigger:  After Task Breakdown completes, or whenever the Dependency Manager detects a slip
Reads:  tasks, task_dependencies, client_config.working_calendar
Writes:  tasks.planned_start / planned_end
Critical Path Method. The same algorithm runs regardless of project or client — only the data changes (Section 2, config vs. code).
1.  Load every task and dependency edge for the project (or the affected phase, if re-running after a slip).
2.  Build a directed graph: tasks as nodes, dependencies as edges.
3.  Topologically sort the graph.
4.  Forward pass: compute the earliest possible start and finish for every task, skipping weekends/holidays per the client's working_calendar.
5.  Backward pass: compute the latest allowable start and finish for every task, working backward from the phase or project end date.
6.  Compute slack per task (latest start minus earliest start).
7.  Identify the critical path — every task with zero slack.
8.  Write the computed planned_start / planned_end back to each task.
9.  If total computed duration exceeds the stated project timeline, raise a Tier 1 review item rather than silently accepting an infeasible plan.
10.  Log the run to audit_log.
8.3  Assignment Engine
  Deterministic — plain code, no LLM  
Trigger:  After the Scheduler produces dates, or whenever a new task appears mid-project
Reads:  tasks, team_members, task assignments across every active project
Writes:  tasks.owner_id, team_members.allocated_hrs
Greedy skills-and-capacity match. The capacity check is cross-project by design (Section 9) — this is the step most likely to silently double-book someone if built carelessly.
1.  Sort tasks needing assignment, critical-path tasks first.
2.  For each task, filter team_members to those whose skill_tags overlap the task's required skills and who are marked active.
3.  For each candidate, compute their true current allocated_hrs by summing committed hours across every active project they're on — not just this one.
4.  Score remaining candidates using the client's configured assignment_strategy (best skill match, or balanced workload).
5.  Assign the top-scored candidate. If nobody has remaining capacity, do not over-allocate — flag the task as unassignable and raise a Tier 1 review item instead.
6.  Write owner_id on the task and update that person's allocated_hrs.
7.  Log the run to audit_log.
8.4  Status Tracking
  Hybrid — EVM math + Sonnet 5  
Trigger:  Client's configured reporting cadence, or an ad hoc self-report
Reads:  Free-text status replies from task owners
Writes:  tasks.status, risks_issues (on threshold breach)
The math is deterministic once a status exists; Sonnet 5's only job is turning a person's free-text reply into that status in the first place.
1.  On the configured cadence, message each task owner asking for a status update on their open tasks.
2.  Receive the free-text reply.
3.  Sonnet 5 parses the reply into a structured status (and percent-complete estimate, if mentioned).
4.  Validate the parsed status against the allowed set of values — flag as ambiguous rather than guessing if the reply doesn't clearly map to one.
5.  Update the task's status and actual_start / actual_end as appropriate.
6.  Deterministic step: recompute Planned Value, Earned Value, and Actual Cost; derive Schedule Variance (EV − PV) and Cost Variance (EV − AC).
7.  Compare both variances against the client's configured slip_threshold_days.
8.  If a threshold is breached, raise a Tier 1 off-track alert and create or update the corresponding risks_issues entry.
9.  Log the cycle to audit_log.
8.5  Risk & Issue Tracking
  Hybrid — rule triggers + Sonnet 5  
Trigger:  Continuous — runs after every status-tracking cycle and every meeting summary
Reads:  Variance data, meeting notes, existing open risks
Writes:  risks_issues
Two independent detection paths feed the same register, with a duplicate check before anything new is added.
1.  Rule-based pass: check deterministic thresholds — schedule variance, cost variance, capacity over-allocation flagged by the Assignment Engine. Any breach auto-creates a risk candidate, no LLM involved.
2.  LLM pass: send recent meeting notes and status free-text to Sonnet 5, asking it to flag anything that reads like an emerging risk the rules above wouldn't catch.
3.  Before inserting either kind of candidate, check it against every existing open risk on the project and ask Sonnet 5 whether it's a duplicate. Skip the insert if so.
4.  Score the new risk (severity × likelihood, 1–5 each) — rule-assigned where possible, Sonnet-suggested otherwise, always reviewer-adjustable.
5.  Insert into risks_issues, tagged with its source (rule_based or pattern_detected).
6.  Raise a Tier 1 review item for reviewer awareness.
7.  Log the cycle to audit_log.
8.6  Dependency Manager
  Deterministic — plain code, no LLM  
Trigger:  A task's actual_end is recorded later than its planned_end
Reads:  task_dependencies, tasks
Writes:  tasks.planned_start / planned_end (downstream tasks only)
Graph traversal, reusing the same dependency graph the Scheduler builds. Sonnet 5's only role is explaining the result in plain language — it never decides the dates.
1.  Detect the slip: an actual completion date later than the planned one.
2.  Compute the slip's magnitude in days.
3.  Walk the dependency graph forward from the slipped task to find every downstream task it affects.
4.  Re-run the Scheduler's forward and backward pass, restricted to the affected tasks.
5.  Diff the new dates against the old ones for every downstream task, and against any phase or project end date.
6.  If a milestone or end date moved by more than the client's slip_threshold_days, raise a Tier 1 review item. Sonnet 5 turns the raw diff into a plain-language summary (e.g. “this pushes the client demo milestone by 2 days”).
7.  Update planned dates on every affected task.
8.  Log the run to audit_log.
8.7  Meeting Summary
  LLM reasoning — Claude Sonnet 5  
Trigger:  A transcript or set of notes is uploaded by an admin or team member
Reads:  Raw transcript / notes text
Writes:  meetings, meeting_action_items, blockers, and (indirectly) tasks
Runs before Task Breakdown and Risk Tracking in each cycle — its output becomes their input, not the other way around.
1.  Receive the raw transcript or notes.
2.  Send it to Sonnet 5 with an instruction to extract exactly three buckets: decisions (with who decided), action items (with owner and due date if stated), and blockers (who's blocked, on what).
3.  Validate the output against that three-bucket structure.
4.  Write the meeting record into meetings, with decisions stored as structured data.
5.  For each action item, create a row in meeting_action_items. If it implies new work, hand it to Task Breakdown to generate a linked task, and record that link (converted_task_id).
6.  For each blocker, create or update a row in blockers with a description and, where statable, who's responsible for resolving it (assigned_to). If ownership isn't clear from the notes, leave it unassigned and flag it to the reviewer rather than guessing.
7.  Log the extraction to audit_log.
8.8  Stakeholder Comms
  LLM reasoning — Claude Sonnet 5  
Trigger:  Client's configured communication cadence, or an ad hoc reviewer request
Reads:  Status variance, open risks, recent milestone changes, stakeholders.audience_type
Writes:  review_queue (Tier 2 draft) — never sends directly
Draft-only by design. No matter how good the draft looks, a human reviews and sends it — this is the one skill that never gets a fully-automatic depth setting.
1.  Pull the latest status variance, open risks, and any recent milestone changes for the project.
2.  Determine the target audience type from stakeholders (team, exec, client, or investor).
3.  Send this data plus the client's configured voice_style to Sonnet 5, with instructions to draft a message suited to that audience — a two-line health check for an executive, feature-progress framing for a client, full technical detail internally.
4.  Write the draft into review_queue as a Tier 2 item, tagged as a comms artifact.
5.  The reviewer reviews and edits as needed, then approves.
6.  Only after explicit approval does a human send it — the agent itself never dispatches outbound communication.
7.  Log the draft-and-approval cycle to audit_log; version the final approved text in artifact_versions.
9. Resource Management Across Projects
Team members are shared across every project a client runs — the same roster, not a separate one per project. This makes cross-project capacity the single most important thing the assignment engine has to get right.
A person's allocated hours are tracked as one running total across every active project, not reset per project. Before assigning any task, on any project, the assignment engine checks that person's true remaining capacity given everything else they are already committed to — not just the current project's task list.
Correction from v3:  Without this, the assignment engine could independently assign a person to full capacity on Project A, then separately assign them to full capacity on Project B, because each run only saw its own project's tasks. This was previously documented only as an edge case (Section 16); it is now a first-order design requirement, since it is the most likely way this system silently produces an unworkable plan.
•  Blockers now distinguish who raised an issue from who is responsible for resolving it (blockers.assigned_to), since these are often different people.

11. Agent Operating Cycle
Project docs submitted (per project) → agent breaks scope into phases, then tasks within each phase → agent schedules and assigns (checking capacity across every active project) → agent runs & monitors (tracking, risk, dependency management, blockers with a clear owner) → reviewer checks in (tap approval, tagged by project) → agent produces outputs (reports, comms, sign-off packets). The middle steps repeat continuously until the project closes, at which point a retrospective is generated and the project is archived.

13. Quality Bar
•  No fabrication — every output cites its basis or explicitly flags low confidence rather than guessing.
•  Every status report, risk register, and sign-off packet is versioned (artifact_versions).
•  Every agent action is audit-logged with timestamp, input, and output (audit_log).
•  Every escalation is logged with timestamp, reason, and outcome (escalation_log).
•  Config edits are validated on every save, not only at initial onboarding.
14. Out of Scope for V1
•  Multi-tenant / multi-client support — deferred; the schema supports adding it later without a redesign.
•  External tools (e.g. Jira) as the system of record — v1 uses its own mini-database.
•  Any form of automated final sign-off or change approval without a human tap, regardless of skill-depth configuration.
15. Acceptance Criteria
•  Deterministic skills (scheduler, assignment engine, dependency manager) produce correct output against a hand-verified test dataset with a known answer.
•  LLM reasoning skills are evaluated against a held-out sample project not used during development.
•  Task breakdown produces phases with valid dates before generating any tasks under them.
•  A team member's combined allocation across every active project never silently exceeds their capacity.
•  No Tier 2 or Tier 3 item is ever auto-approved under any configuration.
•  A silent reviewer triggers backup notification within the configured delay, and a silent backup triggers a paused-work state, verified in a live test.
•  Config edits are validated on every save; a missing required field is surfaced as a defect, not silently accepted.
•  A second, independent project runs correctly on the same shared codebase with zero project-specific code.
16. Edge Cases
•  A reviewer edits client_requirements.md directly and leaves a required field blank — flagged as a config defect, not silently accepted.
•  A project's config_overrides conflicts with the client default in a way that would break a downstream skill (e.g. no reviewer set at either level) — that skill refuses to run for that project until resolved.
•  Backup reviewer is also unset — escalation goes straight to the paused-work state rather than erroring or silently proceeding.
•  A blocker is raised by one person but has no clear owner assigned — surfaced to the reviewer rather than left unassigned indefinitely.
